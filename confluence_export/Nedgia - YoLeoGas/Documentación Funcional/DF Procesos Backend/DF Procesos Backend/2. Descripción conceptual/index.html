2.1. Descripción de la necesidad

Se requieren una serie de procesos para:

  • gestionar todo el ciclo de vida de las lecturas

  • enviar las notificaciones push

  • generar el informe para el Data lake

  • historificar datos

  • eliminar imágenes ya procesadas

2.2. Modelo de sistemas

2.2.1. Sistemas

Los sistemas que intervendrán serán los siguientes:

  • app YoLeoGas → aplicación móvil desde donde se realizan las lecturas de contador

  • OCR → sistema que a partir de una imagen de contador saca la lectura numérica

  • Backoffice → sistema que contiene la información de todas las lecturas que se han realizado desde la app y su ciclo de vida

  • Zeus → sistema de Nedgia donde están registrados todos los CUPS de gas de la distribuidora y sus lecturas

  • Plataforma notificaciones push (Pushwoosh/Firebase)→ sistema que gestiona el envío de las notificaciones push

  • Blob Storage → sistema donde se guardan todas las fotografías de contador

2.2.2. Interfaces

Los procesos de backend hacen que el sistema backoffice se comunique con el OCR de servidor, Zeus y con la plataforma de notificaciones push.

  • Con el OCR de servidor para:

    • Enviar la imagen de contador de gas para que el OCR la procese

    • Obtener la lectura que el OCR ha interpretado

  • Con ZEUS para:

    • Enviar la lectura a Nedgia para que las comercializadoras la consideren en la siguiente facturación

    • Recibir qué CUPS tienen próxima facturación para enviarles una notificación

  • Con la plataforma de notificaciones push para:

    • Envío de las notificaciones push a aquellos CUPS que Zeus ha informado que tendrán una próxima notificación

2.5. Procesos a desarrollar

A continuación se describen a alto nivel los procesos desarrollados y sus funcionalidades.

  • Ciclo de vida lecturas → todas las lecturas realizadas desde la aplicación móvil de YoLeoGas y las que se han validado desde la web de backoffice se envían al OCR de servidor. Las que pasan el índice de confianza del OCR, son enviadas directamente a la distribuidora y las que no, se quedan pendientes de revisión desde la web de backoffice. Con la respuesta de la distribuidora, finaliza el proceso.

  • Envío de notificaciones → la distribuidora Nedgia envía un listado con todos los CUPS para los que finaliza el periodo de facturación y se envía una notificación push a todos los dispositivos asociados a estos CUPS.

  • Informes→ se genera el informe necesario para el datalake y el informe de lecturas mensual.

  • Historificación de los datos→ se pasan a las tablas de histórico los datos con una antigüedad superior a 6 meses.

  • Limpieza de documentos→ se eliminan del Blog Storage todas las imágenes asociadas a lecturas procesadas hace más de 6 meses.
