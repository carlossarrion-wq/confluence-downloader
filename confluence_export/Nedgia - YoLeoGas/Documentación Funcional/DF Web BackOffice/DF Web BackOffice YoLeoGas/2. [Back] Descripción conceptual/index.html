2.1. Descripción de la necesidad

Se requiere una renovación tecnológica de la aplicación web de backoffice de Yo Leo Gas y adicionalmente se han revisado las funcionalidades existentes de forma que:

  • se han eliminado las que afectaban a distribuidoras diferentes de Nedgia

  • la sección de informes ha cambiado, mostrando aquellos informes que se enviaban por email

2.2. Modelo de sistemas

2.2.1. Sistemas

Los sistemas que intervendrán serán los siguientes:

  • app YoLeoGas → aplicación móvil desde donde se realizan las lecturas de contador y se envían

  • EntraId → sistema que verifica la identidad de los usuarios y controla a qué tienen permiso para acceder

  • OCR → sistema que a partir de una imagen de contador saca la lectura numérica

  • Backoffice → sistema que contiene la información de todas las lecturas que se han realizado desde la app y su ciclo de vida

  • Zeus → sistema de Nedgia donde están registrados todos los CUPS de gas de la distribuidora y sus lecturas

  • Plataforma notificaciones push (Pushwoosh/Firebase)→ sistema que gestiona el envío de las notificaciones push

  • Blob Storage → sistema donde se guardan todas las fotografías de contador para que la app YoLeoGas y la web de backoffice puedan consultarlas

  • Documentum → sistema donde se guardan todas las fotografías de contador para que se puedan consultar desde otros sistemas externos

2.2.2. Interfaces

La aplicación web de backoffice se integra directamente con la aplicación móvil Yo Leo Gas, EntraId y el Blog Storage.

  • Con la aplicación móvil YoLeoGas para:

    • Recibir toda la información de las imágenes y lecturas de contador de gas realizadas

  • Con EntraId para:

    • Recibir el nombre, apellidos y rol de la persona que entra en la aplicación

  • Con el Blob Storage

    • Recuperar la imagen de la fotografía realizada del contador

También se integra indirectamente a través de los procesos de backend con el OCR de servidor, con Zeus y con Documentum.

  • Con el OCR de servidor para:

    • Enviar la imagen de contador de gas para que el OCR la procese

    • Obtener la lectura que el OCR ha interpretado

  • Con ZEUS (a través del BUS) para:

    • Enviar la lectura a Nedgia para que las comercializadoras la consideren en la siguiente facturación

    • Recibir qué CUPS tienen próxima facturación para enviarles una notificación

  • Con Documentum para:

    • Guardar la imagen de la fotografía realizada del contador para que sea accesible desde otros sistemas

2.4.2. Descripción de entidades

Las entidades principales que forman parte del modelo de la aplicación web de backoffice de YoLeoGas son:

  • USER → usuarios de la aplicación web de backoffice

    • user_id: identificador único del usuario

    • user_name: nombre y apellidos del usuario

    • user_lastlogin: fecha del último acceso del usuario a la aplicación

    • user_type: rol del usuario

  • PROFILE → perfiles de contadores registrados en el sistema

    • prf_id: identificador único del perfil

    • prf_name: nombre del perfil

    • prf_cups: CUPS del punto de suministro

    • prf_image_blob: nombre de la imagen que se le ha dado al subirla al blob storage

    • prf_image_input_chanel: canal desde el que se ha creado el perfil (valores: MOBILE, HERMES)

    • prf_status_result_server: estado que ha asignado el OCR del servidor al perfil (valores: OK, NOK)

    • prf_status_result_manual: si se ha validado el perfil manualmente, estado que le ha asignado el OCR del servidor tras esta validación manual

    • pfr_status_stage: estado en el que se encuentra el perfil (valores: PENDING_SERVER_PROCESSING, PENDING_IMAGE_PROCESSING,
      PENDING_MANUAL_PROCESSING, PROCESSED, DISCARDED)

    • prf_status_reason_nok_server: motivo del KO del OCR del servidor

    • prf_status_reason_non_manual: motivo del KO del OCR del servidor tras una validación manual del perfil

    • prf_last_reading_date: fecha de la última lectura realizada

    • prf_creation_date: fecha en la que se creó el perfil

    • prf_meter_serial_value_server_id: número de serie del contador que ha identificado el OCR

    • prf_meter_serial_value_manual: número de serie del contador que se ha introducido manualmente

    • prf_date_processed_server: fecha en la que se ha procesado el perfil por el OCR del servidor

    • prf_date_processed_manual: fecha en la que se ha procesado el perfil por el OCR del servidor tras validarlo manualmente

    • prf_meter_serial_confidence_server_id: nivel de confianza que ha devuelto el OCR del servidor

    • oper_id: identificador del operario que ha creado el perfil

    • pr_notification_hour: hora en la que se puede enviar la notiicación push

    • prf_allow_notifications: indica si el usuario quiere recibir notificación push (0=no, 1=sí)

    • prf_locale: idioma del dispositivo

    • prf_deleted: indica si el perfil se ha borrado desde la aplicación móvil

    • prf_manual_supervision: indica si requiere supervisión manual

    • prf_notifications_received: número de notificaciones que se han enviado al CUPS

  • IMAGE → Imágenes de contadores para identificar la lectura del mismo, realizadas desde la app móvil YoLeoGas y W&T

    • image_id: identificador único de la imagen

    • prf_id: perfil del contador al que está asociada la imagen

    • image_blob: nombre de la imagen que se le ha dado al subirla al blob storage

    • image_input_channel: canal desde el que se ha creado la imagen (valores: MOBILE, HERMES,MOBILEPROFILE)

    • image_status_stage: estado de la imagen (valores: PENDING_PROFILE_VALIDATION → pendiente validación perfil, PENDING_SERVER_PROCESSING → pendiente que la procese el OCR del servidor, SERVER_PROCESSING → procesando el servidor, PENDING_MANUAL_PROCESSING-> pendiente de validación manual, PENDING_SGC_PROCESSING → pendiente de envío a la distribuidora, SGC_PROCESSING-> enviado a distribuidora, PROCESSED_OK-> procesado correctamente, PROCESSED_HERMES -> procesado correctamente si image_input_channel es HERMES, DISCARDED-> descartado )

    • image_status_result_mobile: estado de la lectura que ha dado el OCR móvil (valores: OK, NOK)

    • image_status_result_server: estado de la lectura que ha dado el OCR del servidor (valores: OK, NOK)

    • image_status_result_manual: estado de la lectura que no ha llegado al OCR del servidor (valores: vacío, NOK)

    • image_status_result_server_manual: estado de la lectura que ha dado el OCR del servidor tras dar manualmente la lectura

    • image_status_reason_nok_server: motivo del KO del OCR servidor

    • image_status_reason_nok_manual: motivo del KO del OCR servidor tras dar la lectura manual

    • image_status_reason_nok_sgc: código de error que ha devuelto la distribuidora

    • image_date_processed_mobile: fecha en la que se ha recibido la lectura

    • image_date_processed_manual: fecha en la que ha validado manualmente la lectura

    • image_date_processed_sgc: fecha en la que se ha enviado la lectura a la distribuidora

    • user_id: usuario que ha validado manualmente la lectura

    • image_meter_reading_value_user: valor de la lectura que se ha enviado desde la app móvil (la del OCR móvil o la modificada manualmente)

    • image_meter_reading_value_mobil: valor de la lectura que ha enviado el OCR de la app móvil

    • image_meter_reading_value_server: valor de la lectura que ha enviado el OCR servidor

    • image_meter_reading_value_manual: valor de la lectura que se ha dado cuando se ha validado manualmente

    • image_meter_reading_confidence_mobile: nivel de confianza global que ha dado el OCR móvil

    • image_meter_reading_confidence_server: nivel de confianza que ha dado el OCR servidor

    • image_date_processed_documentum: fecha en la que se ha enviado la imagen a Documentum

    • image_retry: número de veces que se ha repetido la fotografía antes de enviarla

    • image_manual_supervision: indica si se ha marcado manualmente para supervisión

    • image_meter_reading_confidence_mobile_digit*: nivel de confianza del OCR de la app móvil para cada dígito (solo para canal MOBILE)

    • image_meter_reading_confidence_server_digit*: nivel de confianza del OCR del servidor para cada dígito

    • platform: sistema operativo del dispositivo desde donde se ha capturado la imagen

    • phone_model: modelo del dispositivo desde donde se ha capturado la imagen

    • app_version: versión de la aplicación móvil desde donde se ha capturado la imagen

    • info_device: versión del sistema operativo del dispositivo desde donde se ha capturado la imagen

    • image_date_processed_server_reading_start: fecha en la que el OCR del servidor ha empezado a procesar la imagen

    • image_date_processed_server_reading_end: fecha en la que el OCR del servidor ha terminado de procesar la imagen

  • OPERATOR → información de los técnicos que capturan las imágenes con dispositivos móviles desde la app móvil W&T

    • oper_id: identificador único del operador

    • oper_oper_id: código de identicación del operador

    • oper_name: nombre y apellidos del operador

    • oper_last_image_date: fecha de la última imagen capturada por el operador

    • oper_last_image_app_version: versión de la aplicación móvil con la que se ha capturado la última imagen

  • NEXTIMAGENOTIFICATION → CUPS que envía Zeus conforme tendrán una próxima facturación y hay que avisar vía notificación push

    • nin_cups: CUPS que envía Zeus

    • nin_date: fecha en la que se ha de enviar la notificación

  • NOTIFICATION -> Notificaciones que se han de enviar a los clientes cuando Zeus informa que hay próxima una facturación

    • notification_id: identificador único de la notificación

    • prf_id: perfil al que corresponde la notificación

    • notification_creation_date: fecha en la que se ha creado la notificación

    • notification_message: texto que se ha enviado en la notificación

    • notification_param1..10 : parámetros que se envían para que personalizar la notificación

    • notification_processing_date: fecha en la que se ha enviado la notificación

  • CONFIGURATION ->Parámetros de configuración del sistema a nivel global

    • cfg_ocrService_mobile_MinOcrConfidenceForDigit1 ..8 : confianza mínima por dígito para la lectura en el OCR del móvil

    • cfg_ocrService_server_MinOcrConfidenceForDigit1..8 : confianza mínima por dígito para la lectura en el OCR servidor

    • cfg_profileService_MinOCRConfidenceForIdDigit: confianza mínima dígito para el perfil en el OCR servidor

    • cfg_profileService_MinOCRGlobalConfidenceForIdDigit: confianza mínima global para el perfil en el OCR servidor

    • cfg_notifier_execution_hour: hora de ejecución del proceso de envío de notificaciones

    • cfg_documentum_retain_image_days: días que se guardan las imágenes en el Storage de Azure

    • cfg_excess_reports_hour: hora a la que se envía el email si hay exceso de consumo

    • cfg_excess_reports_destination: emails a los que se se enviará el informe de lecturas con exceso de consumo

    • cfg_backoffice_report_hour: hora informe backoffice

    • cfg_minutes_between_distributor_retries: minutos entre reintentos distribuidor

    • cfg_months_historical_data: meses que hay que tener en cuenta para pasar datos a las tablas de histórico

    • cfg_reindexer_hour: horas en las que se ejecuta el proceso de pasar los datos a las tablas de histórico

    • user_id: identificador del usuario de la web de backoffice que realiza la modificación de algún parámetro

    • cfg_last_update: fecha de actualización

  • DISTRIBUTOR_CONFIGURATION ->Configuraciones específicas para la distribuidora

    • dis_id: identificador de la distribuidora

    • dconfig_integration_url: url del endpoint donde se enviarán las lecturas recibidas en YoLeoGas para la distribuidora

    • dconfig_ocrService_server_MinOcrConfidenceForDigitTolerant1..8 : confianza mínima por dígitos en OCR servidor restrictivo

    • dconfig_ocrService_server_MinOcrConfidenceForDigitRestrictive1..8: confianza mínima por dígitos en OCR servidor tolerante

    • dconfig_notifications_1st_message_es / _ca _en /_gl : mensaje de la primera notificación del envío de lectura

    • dconfig_backoffice_report_emails: emails de actividad del backoffice

    • dconfig_readings_monthly_report_emails: emails para informe mensual YoLeoGas

    • user_id: usuario que ha realizado la actualización

    • dconfig_date: fecha de actualización

  • GLOBAL_CONFIGURATION → Configuraciones globales

    • datalake_fecha_procesado: última fecha en la que se ha procesado el informe para el Data Lake

    • datalake_nombre_fichero: nombre del informe del Data Lake

    • datalake_nombre_bucket: nombre del bucket de Nedgia donde se ubica el informe del Data Lake

    • datalake_cabecera: cabecera del informe del Data Lake

    • datalake_ruta_bucket: ruta dentro del bucket de Nedgia donde se deja el informe del Data Lake

    • datalake_clave_acceso: clave de acceso para la comunicación con el bucket de Nedgia donde se deja el informe del Data Lake

    • datalake_clave_acceso_secreta : clave de acceso encriptada para la comunicación con el bucket de Nedgia donde se deja el informe del Data Lake

    • datalake_hora: hora en la que se genera el informe del Data Lake

    • datalake_campos: campos del informe del Data Lake

    • datalake_distribuidora: distridora a considerar en el informe del Data Lake

    • datalake_dia_semana: día de la semana en el que se genera el informe del Data Lake

    • cantidad_registros_informe_mensual

  • IMAGE_HIST -> Registro histórico de IMAGE, tiene la misma estructura que IMAGE

  • PROFILE_HIST -> Registro histórico de PROFILE, tiene la misma estructura que PROFILE

  • NOTIFICATION_HIST -> Registro histórico de NOTIFICATION, tiene la misma estructura que NOTIFICATION

  • WMB → Lecturas que se han enviado a distribuidora

  • LOGS → Logs de procesos backend

2.5. Procesos a desarrollar

A continuación se describen a alto nivel los procesos desarrollados y sus funcionalidades.

  • Validación de lecturas → todas las lecturas que no han pasado el proceso de OCR del servidor no son enviadas a la distribuidora y el usuario puede consultarlas para introducir manualmente la lectura correcta basándose en la información que tiene disponible: imagen del contador, lectura OCR aplicación móvil y lectura OCR servidor.

  • Validación de perfiles → todos los perfiles incompletos (porque el proceso de OCR no ha podido detectar el número de serie) son reportados y el usuario puede consultarlos y validarlos introduciendo manualmente el número de serie a partir de la imagen del contador que tiene disponible.

  • Lecturas → el usuario puede consultar todas las lecturas recibidas y su estado

  • Perfiles → el usuario puede consultar todos los perfiles y su estado

  • Consulta de usuarios → los usuarios con rol Administrador pueden consultar todos los usuarios que acceden a la aplicación y su rol

  • Informes → el usuario puede consultar los informes definidos

Los procesos de backend se especificarán en el diseño funcional DF Backend.
