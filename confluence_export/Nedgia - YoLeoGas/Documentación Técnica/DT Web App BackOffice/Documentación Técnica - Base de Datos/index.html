Modelo Conceptual de Datos

Visión General del Sistema

El sistema YoleoGas gestiona el proceso completo de lectura de contadores de gas, desde la captura de imágenes por parte de técnicos de campo hasta la validación manual por operadores de backoffice. El modelo de datos está diseñado para soportar:

  • Procesamiento Multi-etapa: Mobile → OCR Server → Manual → SGC

  • Multi-tenancy: Soporte para múltiples distribuidores de gas

  • Trazabilidad Completa: Auditoría de todos los cambios y procesos

  • Gestión de Perfiles: Configuración de contadores y modelos OCR

  • Supervisión Jerárquica: Diferentes niveles de acceso y validación

Conceptos Fundamentales

1. Lectura de Contador (Image)

Representa una imagen de contador de gas capturada por un técnico, que pasa por múltiples etapas de procesamiento:

  • Mobile: Procesamiento inicial en dispositivo móvil

  • Server: Procesamiento OCR en servidor

  • Manual: Validación manual por operador

  • SGC: Integración con sistema de gestión comercial

2. Perfil de Contador (Profile)

Configuración específica de un contador de gas que incluye:

  • Información del CUPS (Código Universal del Punto de Suministro)

  • Modelo de contador asociado

  • Historial de lecturas

  • Configuración de notificaciones

3. Modelo OCR (Model)

Definición de las características visuales de un tipo de contador para el procesamiento OCR:

  • Coordenadas de regiones de interés (contador, serie, código de barras)

  • Parámetros de segmentación

  • Configuración de dígitos

4. Distribuidor (Distributor)

Empresa distribuidora de gas que utiliza el sistema, permitiendo multi-tenancy.

Explicación del Proceso de Validación de Perfiles

El proceso de validación de perfiles en YoleoGas es un flujo crítico que asegura la correcta configuración de cada contador de gas antes de que pueda procesar lecturas. Este proceso se ejecuta en varias etapas:

  1. Validación Inicial

  • Se verifica que el código CUPS sea válido y único en el sistema

  • Se validan los datos básicos del perfil (nombre, distribuidor, etc.)

  • Si hay errores en esta etapa, el perfil es rechazado inmediatamente

  1. Asignación de Modelo OCR

  • El sistema busca un modelo OCR compatible basado en las características del contador

  • Si no se encuentra un modelo automáticamente, el perfil queda en estado PENDING_MODEL_ASSIGNMENT

  • Se valida la compatibilidad entre el modelo seleccionado y el tipo de contador

  1. Procesamiento OCR

  • Se ejecuta el procesamiento OCR para extraer el número de serie del contador

  • Si la extracción falla o tiene baja confianza, pasa a validación manual

  • Se aplican las reglas de negocio específicas del distribuidor

  1. Supervisión y Validación Manual

  • Los perfiles que requieren supervisión (prf_manual_supervision = true) son asignados a supervisores

  • Los perfiles estándar pueden ser validados por operadores regulares

  • Se registra toda la actividad en PROFILE_HIST para auditoría

  1. Estados Finales

  • PROCESSED_OK: Perfil completamente validado y operativo

  • PENDING_MODEL_ASSIGNMENT: Esperando asignación de modelo OCR

  • PENDING_MANUAL_PROCESSING: Requiere intervención manual

  • REJECTED: Perfil rechazado por errores críticos

Características Clave del Proceso:

  • Trazabilidad Completa: Cada cambio se registra en PROFILE_HIST

  • Notificaciones Automáticas: Se envían alertas según el estado del perfil

  • Supervisión Jerárquica: Diferentes niveles de validación según la complejidad

  • Recuperación de Errores: Posibilidad de reintento y corrección manual

  • Multi-tenancy: Reglas específicas por distribuidor aplicadas automáticamente

Este proceso garantiza que solo los perfiles correctamente configurados puedan generar lecturas válidas, manteniendo la integridad del sistema y la calidad de los datos procesados.

Explicación del Proceso de Validación de Lecturas

El proceso de validación de lecturas en YoleoGas es el flujo más crítico del sistema, transformando imágenes capturadas en lecturas válidas a través de múltiples etapas de procesamiento y validación:

  1. Captura y Validación Inicial

  • El técnico captura una imagen del contador usando la aplicación móvil

  • Se crea un registro IMAGE con estado inicial y metadatos del dispositivo

  • Se valida la integridad de la imagen y su asociación con un perfil válido

  1. Procesamiento Mobile

  • La aplicación móvil realiza un procesamiento inicial de la imagen

  • Se extrae una lectura preliminar (image_meter_reading_value_mobile)

  • Se calcula un nivel de confianza para cada dígito (image_meter_reading_confidence_mobile_digit*)

  • Si la confianza es baja, se marca para procesamiento adicional en servidor

  1. Procesamiento OCR Server

  • El servidor ejecuta algoritmos OCR avanzados sobre la imagen

  • Se extrae la lectura del servidor (image_meter_reading_value_server)

  • Se calculan niveles de confianza detallados por dígito (image_meter_reading_confidence_server_digit*)

  • Se valida la coherencia con el perfil del contador y lecturas anteriores

  1. Validación de Reglas de Negocio

  • Se aplican reglas específicas del distribuidor (distributor_model_rules)

  • Se verifican rangos válidos y patrones de consumo

  • Se detectan anomalías o lecturas sospechosas

  • Se determina si la lectura requiere supervisión manual

  1. Procesamiento Manual

  • Las lecturas con baja confianza o que fallan validaciones automáticas pasan a validación manual

  • Se asignan a operadores regulares o supervisores según image_manual_supervision

  • Los operadores pueden aceptar, corregir o rechazar la lectura

  • Se registra la decisión en image_meter_reading_value_manual

  1. Gestión de Reintentos

  • Las lecturas rechazadas pueden reintentarse hasta un máximo configurado

  • Se incrementa image_retries_count en cada intento

  • Después del máximo de reintentos, la lectura se marca como definitivamente rechazada

  1. Envío a SGC (Sistema de Gestión Comercial)

  • Las lecturas validadas se preparan para envío al sistema comercial

  • Se realiza una validación final de completitud de datos

  • Se envía al SGC y se registra la respuesta

  • En caso de éxito, se actualiza el perfil con la nueva lectura

  1. Auditoría y Trazabilidad

  • Cada cambio de estado se registra en IMAGE_HIST

  • Las actualizaciones de perfil se registran en PROFILE_HIST

  • Se mantiene un registro completo de la cadena de procesamiento

Estados Principales del Proceso:

  • MOBILE_PROCESSED: Procesamiento móvil completado

  • PENDING_MANUAL_PROCESSING: Requiere validación manual

  • MANUAL_PROCESSED: Validación manual completada

  • PROCESSED_OK: Procesamiento automático exitoso

  • SENT_TO_SGC: Enviado al sistema comercial

  • REJECTED: Lectura rechazada definitivamente

Características Clave del Proceso:

  • Procesamiento Multi-etapa: Mobile → OCR → Manual → SGC

  • Validación Jerárquica: Operadores y supervisores con diferentes permisos

  • Gestión de Confianza: Niveles de confianza por dígito para decisiones automáticas

  • Recuperación de Errores: Sistema de reintentos y escalado a supervisión

  • Trazabilidad Completa: Auditoría de cada paso del proceso

  • Notificaciones Inteligentes: Alertas contextuales según el estado

  • Multi-tenancy: Reglas específicas por distribuidor aplicadas automáticamente

Este proceso asegura que las lecturas de contadores mantengan la máxima precisión y confiabilidad, con múltiples capas de validación y la flexibilidad para manejar casos excepcionales a través de intervención humana cuando sea necesario.

Consideraciones de Diseño

Escalabilidad

  • Uso de GUIDs para claves primarias (distribución en múltiples servidores)

  • Particionamiento por distribuidor (dis_id) para multi-tenancy

  • Tablas históricas separadas para mantener performance en tablas principales

Integridad Referencial

  • Relaciones opcionales (nullable FKs) para flexibilidad en el flujo de procesamiento

  • Soft deletes mediante campos de estado en lugar de eliminación física

  • Auditoría completa mediante tablas *_hist

Performance

  • Índices optimizados para consultas de backoffice (estado, fecha, distribuidor)

  • Separación de datos históricos para reducir el tamaño de tablas activas

  • Campos desnormalizados para evitar JOINs complejos en consultas frecuentes

Conclusiones

El modelo de datos de YoleoGas está diseñado para soportar un sistema empresarial complejo de gestión de lecturas de contadores de gas con las siguientes características clave:

  1. Multi-tenancy: Soporte nativo para múltiples distribuidores

  2. Trazabilidad Completa: Auditoría de todos los cambios y procesos

  3. Escalabilidad: Diseño preparado para millones de lecturas

  4. Flexibilidad: Configuración granular por distribuidor y tipo de contador

  5. Integridad: Relaciones bien definidas con validación de datos

El sistema maneja eficientemente el flujo completo desde la captura móvil hasta la integración con sistemas comerciales, manteniendo la integridad de los datos y proporcionando la trazabilidad necesaria para un entorno empresarial regulado.
