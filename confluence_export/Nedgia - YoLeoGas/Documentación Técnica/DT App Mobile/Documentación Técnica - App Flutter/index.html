Resumen Ejecutivo

YoLeoGas es una aplicación móvil desarrollada en Flutter para la lectura automatizada de contadores de gas mediante tecnología OCR (Reconocimiento Óptico de Caracteres) e Inteligencia Artificial. La aplicación permite a los usuarios capturar imágenes de contadores de gas y obtener lecturas precisas automáticamente, sincronizando los datos con el backend de Nedgia.

1. Tecnología Seleccionada

1.1 Framework Principal

  • Flutter 3.8+: Framework multiplataforma de Google para desarrollo móvil

  • Dart: Lenguaje de programación principal

  • SDK mínimo: Android API 21+ (Android 5.0)

1.2 Arquitectura de la Aplicación

  • Patrón BLoC/Cubit: Gestión de estado reactiva con flutter_bloc

  • Arquitectura en Capas:

    • Presentación: Views, Widgets, Cubits

    • Dominio: Modelos de datos, Casos de uso

    • Datos: Servicios, APIs, Base de datos local

1.3 Tecnologías de Inteligencia Artificial

  • TensorFlow Lite: Framework de ML optimizado para móviles

  • Modelos personalizados:

    • ylg_id.tflite: Detección de ID de contadores

    • ylg_reader.tflite: Lectura de valores numéricos

  • OCR Híbrido: Combinación de ML local y procesamiento en servidor

2. Modelo de Datos

2.1 Estructura Principal de Datos

Profile (Contador)

wide760

Estados del contador (ProfileState):

  • NO_PHOTO (0): Sin fotografía

  • PENDING_TO_SEND (1): Pendiente de envío

  • SENT (2): Enviado al servidor

  • PROCESSED (3): Procesado correctamente

  • DISCARDED (4): Descartado

  • TO_DELETE (5): Marcado para eliminación

ImageRecord (Tabla Images)

wide760

Estados de la imagen (ImageRecord.state):

  • PENDING_TO_SEND: Pendiente de envío al servidor

  • SENT: Enviada al servidor, esperando procesamiento

  • PROCESSED: Procesada correctamente por el servidor

  • REJECTED: Rechazada por el servidor

  • FAILED: Error en el envío o procesamiento

  • RETRY: Marcada para reintento automático

Flujo de Estados:

  1. Captura: PENDING_TO_SEND → se almacena localmente con datos OCR

  2. Envío: SENT → se envía al servidor con FormData

  3. Procesamiento: PROCESSED → servidor confirma procesamiento exitoso

  4. Alternativo: REJECTED/FAILED → manejo de errores y reintentos

2.2 Propósito de la Base de Datos Local

Funcionalidad Offline-First

  • Almacenamiento persistente: Todos los datos se guardan localmente primero

  • Funcionamiento sin conexión: La app funciona completamente offline

  • Queue de sincronización: Las operaciones se encolan para envío posterior

  • Historial completo: Mantiene registro de todas las lecturas y estados

Datos Almacenados Localmente

  • Perfiles de contadores: Información de contadores registrados

  • Imágenes de lecturas: Fotos capturadas con metadatos OCR

  • Estados de sincronización: Control de qué se ha enviado al servidor

  • Configuración de usuario: Preferencias y configuraciones

  • Cache de respuestas: Datos recibidos del servidor

8. Ventajas del Modelo Local-First

8.1 Experiencia de Usuario

✅ Funcionamiento offline: App usable sin conexión a internet
✅ Respuesta inmediata: No se esperan respuestas de red para usar la app
✅ Historial completo: Acceso a todas las lecturas anteriores sin conexión
✅ Sincronización transparente: El usuario no gestiona manualmente la sincronización

8.2 Robustez Técnica

✅ Tolerancia a fallos: Los datos no se pierden por problemas de red
✅ Reintentos automáticos: Sincronización resiliente con backoff exponencial
✅ Estados consistentes: Control preciso del flujo de datos y transiciones
✅ Performance: Acceso rápido a datos locales desde SQLite

8.3 Eficiencia de Red

✅ Uso optimizado: Solo sincroniza cuando es necesario
✅ Batch operations: Agrupa operaciones para mayor eficiencia
✅ Compression: Optimización de transferencia de imágenes
✅ Delta sync: Solo cambios, no datos completos

8.4 Consideraciones de Implementación

Consistencia de Datos

  • Single source of truth: El servidor es la autoridad final

  • Conflict resolution: Estados locales se actualizan con respuesta del servidor

  • Transacciones: Operaciones atómicas en base de datos local

Gestión de Errores

  • Network errors: Reintentos automáticos con backoff exponencial

  • Server errors: Manejo específico por código de error HTTP

  • Data corruption: Validación y recuperación de datos

Performance

  • Database indexing: Consultas optimizadas por fecha y estado

  • Memory management: Carga lazy de imágenes grandes

  • Background processing: Sincronización sin bloquear UI
